# Prepare Command

为任意 API 或工具创建结构化知识文档。知识文档是后续 Compile 命令生成环境特定 SKILL.md 的输入。

## 依赖文件（执行前加载）

| 文件 | 何时加载 | 必须 | 用途 |
|------|---------|------|------|
| [references/knowledge-doc-spec.md](references/knowledge-doc-spec.md) | 开始前 | ✅ | 知识文档的标准章节和格式定义 |
| [references/interview-guide.md](references/interview-guide.md) | Phase 2 开始时 | ✅ | 结构化访谈问题库、优先级、引导技巧 |

## 工作模式

**创建模式**: 用户提供 API 配置或文档 → AI 探索 + 访谈 → 输出知识文档
**追加模式**: 用户提供已有知识文档 → AI 识别缺失章节 → 补充

判断方式：用户是否指定了已有知识文档路径？
- 是 → 追加模式（读取已有文档，只补充缺失/标记为 TODO 的部分）
- 否 → 创建模式（从零开始）

## 执行流程

```
Phase 1: API 探索（自动） → 填充 Tool Interface + Request Specification
Phase 2: 用户访谈（交互） → 填充 Iron Rules + Semantic Framework + 其余章节
Phase 3: 知识结构化（自动） → 组装完整知识文档
```

---

### Phase 1: API 探索

目标：自动发现 API 的可用工具、参数和返回格式。

**步骤：**

1. **定位 API 信息源**
   - 用户指定了配置文件？→ 读取，提取 URL 和认证信息
   - 用户指定了 API 文档 URL？→ 抓取并解析
   - 用户指定了 OpenAPI/Swagger spec？→ 直接解析端点列表
   - 都没有？→ 询问用户 API 基础 URL 和认证方式

2. **探索端点**
   - 如有 OpenAPI spec：解析所有端点、参数、响应 schema
   - 如无文档：尝试以下常见端点发现策略
     - `GET /api` 或 `GET /api/v1` — 查看是否有端点列表
     - `GET /swagger.json` 或 `GET /openapi.json` — 查看是否有 spec
     - `GET /health` 或 `GET /status` — 确认 API 是否可达
   - 对发现的每个端点：尝试调用（用最小参数）→ 记录响应格式

3. **整理工具清单**
   对每个可用端点，记录：
   - 工具名称（英文标识）
   - 用途（一句话）
   - 参数列表（名称、类型、是否必填、描述）
   - 返回格式（结构和示例）
   - 返回解析规则（哪个字段在哪里、用于什么）

4. **向用户确认**
   展示发现的工具清单，询问：
   - "这些工具是否完整？有没有我遗漏的？"
   - "这些参数描述是否准确？"

> 如果 API 探索失败或 API 不是 REST 风格，跳过自动探索，在 Phase 2 中向用户逐项询问工具信息。

---

### Phase 2: 用户访谈

目标：收集 AI 无法自动发现的领域知识（铁律、语义规则、业务约束）。

**关键原则：**
- 根据 Phase 1 已获取的信息，跳过已有答案的问题
- 按优先级分轮提问，用户可随时说"先这样"跳过后续轮次
- 每轮结束时总结已收集的内容，让用户确认

**按优先级提问（详见 [references/interview-guide.md](references/interview-guide.md)）：**

**P0（必须，生成最小可用知识文档）：**
- 领域一句话描述（如未从 Phase 1 获取）
- 目标用户是谁
- 3-5 个典型用户表达
- API 请求体格式和必填参数（如 Phase 1 未完全覆盖）
- 结果展示方式

**P1（推荐，显著提升质量）：**
- 硬性规则（"有没有绝对不能违反的规则？"）
  - 引导方式："使用这个 API 时，有哪些做法会导致结果错误但不报错？"
  - 对每条规则追问正确/错误示例
- 语义消歧规则（"用户说的话中，有没有容易混淆的表达？"）
  - 引导方式："同一个词在不同上下文是否有不同含义？"
- 常见错误模式（"新手最常犯的错误是什么？"）

**P2（可选，锦上添花）：**
- 高级使用场景和对应示例
- 参数速查表（常用表达 → API 参数的直接映射）
- 默认策略（用户省略某参数时该怎么推断）
- 操作组合约束（哪些操作不能叠加）
- 筛选条件路由（筛选条件放在哪个位置）
- 输出翻译规则（技术参数如何转为业务语言）

**每轮结束时：**
```
"到目前为止，我已经收集了以下信息：
✅ 领域概述
✅ 3 个工具定义
✅ 5 条硬性规则
⬜ 语义消歧规则（尚未收集）
⬜ 完整示例（尚未收集）

要继续补充还是先生成知识文档？"
```

---

### Phase 3: 知识结构化

目标：将 Phase 1+2 收集的信息组装为知识文档。

**步骤：**

1. **读取格式规范**
   阅读 [references/knowledge-doc-spec.md](references/knowledge-doc-spec.md) 获取知识文档的标准格式。

2. **填充 YAML 元数据**
   从收集的信息中提取：domain, domain_slug, one_liner, target_users, language, api 信息, tools 列表。

3. **逐章节填充**
   按知识文档的 8 个标准章节顺序填充：
   - Domain Overview
   - Tool Interface
   - Iron Rules（如收集了）
   - Semantic Framework（如收集了）
   - Request Specification
   - Output Specification
   - Examples（至少 3 个端到端示例）
   - Error Patterns（如收集了）

4. **标记未完成章节**
   对于用户跳过的章节，用 `<!-- TODO: 待补充 -->` 标记，不留空。

5. **输出知识文档**
   保存到用户指定路径或默认路径 `{project}/knowledge-docs/{domain_slug}.md`

6. **生成摘要**
   向用户展示：
   - 知识文档路径
   - 已完成章节 / 待补充章节
   - 建议下一步："可以直接用 Compile 命令生成 SKILL.md，或继续补充知识文档"

## 追加模式

当用户指定已有知识文档时：

1. 读取已有文档
2. 扫描每个章节：
   - 内容为空或含 `<!-- TODO -->` → 标记为待补充
   - 内容存在 → 跳过（不覆盖）
3. 对待补充章节，按 Phase 2 的方式逐项询问
4. 将新收集的内容追加到对应章节
5. 保存更新后的文档

---

## 质量保障

生成知识文档后，执行以下自检：

- [ ] YAML 元数据的必填字段完整？
- [ ] 至少有 1 个工具定义？
- [ ] 至少有 3 个端到端示例？
- [ ] 每条 Iron Rule 都有 ✅ 和 ❌ 对比示例？
- [ ] Tool Interface 中每个工具都有调用示例和返回格式？
- [ ] 知识文档可被人类独立审阅（无需其他上下文即可理解）？

如有未通过项，在摘要中提示用户。
