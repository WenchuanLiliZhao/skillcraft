# 知识文档格式规范 (Knowledge Document Specification)

> 知识文档是 Prepare 命令的**输出**、Compile 命令的**输入**。
> 它记录了生成一个 API 查询类 Skill 所需的全部领域知识。
> 设计原则：**人可审阅，机可解析**。

---

## 元数据头 (YAML Frontmatter)

```yaml
---
domain: ""                    # [必填] 领域名称，如 "指标查询""SQL 查询"
domain_slug: ""               # [必填] 英文标识，用作生成的 skill name，如 "metric-query"
one_liner: ""                 # [必填] 一句话描述 skill 做什么
target_users: ""              # [必填] 目标用户，如 "业务分析师""数据工程师"
language: ""                  # [必填] 用户交互语言，如 "zh-CN""en"

api:
  base_url: ""                # [选填] API 基础 URL
  auth_method: ""             # [选填] 认证方式描述，如 "Bearer Token""API Key in header"
  request_format: ""          # [必填] 请求体格式，如 "JSON""GraphQL""SQL"

tools:                        # [必填] 工具列表（抽象定义，不含环境特定调用方式）
  - name: ""                  #   工具名称（英文标识）
    display_name: ""          #   展示名称
    purpose: ""               #   一句话说明用途
    params:                   #   参数列表
      - name: ""
        type: ""              #   string / number / boolean / json
        required: true
        description: ""
    returns: ""               #   返回格式描述

created_by: ""                # [自动] prepare-command / manual
created_at: ""                # [自动] ISO 日期
version: 1                    # [自动] 版本号
---
```

---

## 章节定义

知识文档包含以下 **9 个章节**，标题必须严格使用下方的固定命名（`## ` 开头），以便 Compile 解析。

---

### 章节 1: `## Domain Overview`

| 属性 | 值 |
|------|---|
| 必填 | 是 |
| 格式 | 自由文本，3-10 行 |
| 目的 | 让审阅者和 Compile 命令快速理解领域全貌 |

**需要包含：**
- 这个 API/系统做什么（2-3 句）
- 典型用户场景（列举 3-5 个用户会说的话）
- 工作流概述（用户说话 → AI 做什么 → 返回什么）

---

### 章节 2: `## Tool Interface`

| 属性 | 值 |
|------|---|
| 必填 | 是 |
| 格式 | 每个工具一个子节，包含调用示例和返回格式 |
| 目的 | 定义 AI 可用的工具、参数、返回格式的解析方式 |

**每个工具的子节结构：**

```markdown
### {tool_display_name} (`{tool_name}`)

**用途**: {一句话}

**参数**:
- `{param1}` ({type}, 必填): {描述}
- `{param2}` ({type}, 选填): {描述}

**调用示例**:
- {场景1}: `{param1}: "{值}", {param2}: {值}`
- {场景2}: `{param1}: "{值}"`

**返回格式**:
```
{返回内容的格式说明和示例}
```

**返回解析规则**:
- {字段1} 在返回的 {位置}，用于 {用途}
- {字段2} 在返回的 {位置}，用于 {用途}
```

> 注意：此处只描述"调用什么、传什么、返回什么"，不描述"如何调用"（那是环境适配器的事）。

---

### 章节 3: `## Iron Rules`

| 属性 | 值 |
|------|---|
| 必填 | 否（强烈推荐） |
| 格式 | 编号列表，每条规则使用固定子结构 |
| 目的 | 记录"违反即错误"的硬性约束 |

**每条规则的格式 (子模式 P1)：**

```markdown
### 规则 {N} — {规则标题}

{规则描述，1-3 句话说明什么情况下适用}

✅ 正确: {正确做法}
```
{正确示例代码/写法}
```

❌ 错误: {错误做法}
```
{错误示例代码/写法}
```

⚠️ 易错场景: {可选，特别容易踩坑的子场景补充}
```

**写作指南：**
- 数量通常 5-15 条，按重要性/出错频率排序
- 每条必须有 ✅ 和 ❌ 对比示例
- 规则应来自实际使用中发现的错误，而非理论推导
- 如果此领域没有硬性规则，可以只写一条"兜底规则"或留空

---

### 章节 4: `## Semantic Framework`

| 属性 | 值 |
|------|---|
| 必填 | 否（推荐） |
| 格式 | 分为两个子节：分解维度 + 消歧规则 |
| 目的 | 指导 AI 如何理解用户的自然语言请求 |

**子节 4a: `### Decomposition Dimensions` (子模式 P2)**

```markdown
将用户请求分解为以下维度：

| 维度 | 核心问题 | 对应 API 参数 | 示例 |
|------|----------|-------------|------|
| {维度1} | {用户要什么？} | {param_name} | "{用户表达示例}" |
| {维度2} | {怎么处理？} | {param_name} | "{用户表达示例}" |
```

**子节 4b: `### Disambiguation Rules` (子模式 P3)**

```markdown
### 规则 {字母} — {歧义点标题}

| 用户说 | 含义 | 实现方式 |
|--------|------|---------|
| "{表达1}" | {含义A} | {API 做法A} |
| "{表达2}" | {含义B} | {API 做法B} |
| "{表达3}"（无修饰） | 歧义，默认 {X} | {做法X} |
```

**写作指南：**
- 消歧规则数量通常 5-15 条
- 重点关注：同一个词在不同上下文含义不同的情况
- 每条规则应给出"无修饰时的默认选择"

---

### 章节 5: `## Request Specification`

| 属性 | 值 |
|------|---|
| 必填 | 是 |
| 格式 | 分为多个子节，涵盖参数定义、速查表、默认策略等 |
| 目的 | 完整定义 API 请求体的每个参数 |

**包含以下子节：**

**5a: `### Parameters`** — 逐参数定义

```markdown
#### {param_name} ({必填/选填}) — {中文名}

{参数描述，1-2 句话}

**格式**: {JSON 数组 / 字符串 / 对象 / ...}
**语法**: `{语法模板}`
**约束**: {约束条件}

**子特性** (如有):
| 子特性 | 语法 | 含义 | 示例 |
|--------|------|------|------|
| ... | ... | ... | ... |
```

**5b: `### Quick Reference Tables`** (子模式 P4)

```markdown
| 用户说 | {param_name} 值 |
|--------|----------------|
| "{表达}" | `{API 参数值}` |
```

**5c: `### Default Strategies`** (子模式 P5)

```markdown
当用户未指定 {参数} 时：

| 优先级 | 条件 | 默认值 | 原因 |
|--------|------|--------|------|
| 1 | {条件} | {默认值} | {原因} |
```

**5d: `### Filter Routing`** (子模式 P6，如适用)

```markdown
| 场景 | 放在 | 原因 |
|------|------|------|
| {场景描述} | {位置} | {原因} |
```

**5e: `### Composition Constraints`** (子模式 P7，如适用)

```markdown
| 操作A | 操作B | 是否可组合 | 正确做法 |
|-------|-------|-----------|---------|
| ... | ... | ✅/❌ | ... |
```

---

### 章节 6: `## Output Specification`

| 属性 | 值 |
|------|---|
| 必填 | 是 |
| 格式 | 输出结构定义 + 翻译规则表 |
| 目的 | 定义如何向用户展示结果 |

**需要包含：**

```markdown
### Output Structure

输出分为 {N} 个部分，顺序固定：
1. {部分1名称} — {描述和格式要求}
2. {部分2名称} — {描述和格式要求}
3. {部分3名称} — {描述和格式要求}

### Translation Rules (子模式 P8)

| 技术对象 | 翻译为 | 示例 |
|----------|--------|------|
| {API 参数/字段} | {业务语言} | `{技术}` → "{业务}" |
```

---

### 章节 7: `## Examples`

| 属性 | 值 |
|------|---|
| 必填 | 是（至少 3 个） |
| 格式 | 每个示例包含：场景描述、用户输入、完整请求体、预期输出 |
| 目的 | 提供端到端的参考样本 |

**每个示例的格式：**

```markdown
### 示例 {N}: {场景标题}（{难度: 简单/中等/复杂/边界}）

**用户说**: "{自然语言请求}"

**请求体**:
```json
{ ... }
```

**要点**: {此示例展示了什么关键知识点，如"占比 + resultFilters 的正确用法"}
```

**写作指南：**
- 至少覆盖：简单(2-3个) + 中等(3-5个) + 复杂(3-5个) + 边界(2-3个)
- 每个示例应重点展示 1-2 个知识点
- 边界示例应覆盖 Iron Rules 中的易错场景

---

### 章节 8: `## Error Patterns`

| 属性 | 值 |
|------|---|
| 必填 | 否（强烈推荐） |
| 格式 | 编号列表，每条包含：现象、原因、对应规则、正确做法 |
| 目的 | 列出已知的易错场景 |

**每条错误模式的格式：**

```markdown
### ❌ 模式 {N}: {错误名称}

**现象**: {用户/AI 会看到什么异常结果}
**原因**: {为什么会出错}
**对应规则**: 规则 {X} / 无
**正确做法**: {如何避免}
```

**写作指南：**
- 每条 Iron Rule 至少对应一个 Error Pattern
- 额外补充实践中发现的、不属于任何 Iron Rule 的错误
- 按出现频率排序

---

## 章节总览

| # | 章节标题 | 必填 | 对应骨架章节 | 对应子模式 |
|---|---------|------|------------|-----------|
| 1 | Domain Overview | ✅ | #1 概述 | — |
| 2 | Tool Interface | ✅ | #2 工具定义 + #5 工具使用模式 | — |
| 3 | Iron Rules | 推荐 | #3 硬性规则 | P1 |
| 4 | Semantic Framework | 推荐 | #4 语义理解框架 | P2 + P3 |
| 5 | Request Specification | ✅ | #6 请求格式规范 | P4 + P5 + P6 + P7 |
| 6 | Output Specification | ✅ | #8 输出展示规范 | P8 |
| 7 | Examples | ✅ | #9 完整示例 | — |
| 8 | Error Patterns | 推荐 | #10 常见错误模式 | — |

> 骨架中的 #7 构建流程 由 Compile 命令根据上述内容自动生成，不需要在知识文档中手工编写。
