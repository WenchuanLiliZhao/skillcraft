# 访谈指南 (Interview Guide)

本文档定义了 Prepare 命令在 Phase 2 中使用的结构化访谈问题库。

---

## 访谈原则

1. **不问已知**: Phase 1 API 探索已获取的信息，不再询问
2. **从具体到抽象**: 先问具体用法，再问抽象规则
3. **用例驱动**: 让用户举例子比直接问规则更有效
4. **允许跳过**: 每轮结束后给用户"先这样"的选择
5. **循环追问**: 对每条规则/错误，追问"还有吗？"直到用户说没有了

---

## P0: 基础信息（必须收集）

> 目标：收集到足够生成最小可用知识文档的信息。
> 预计耗时：3-5 分钟

### Q1: 领域概述

```
"这个 API/工具是用来做什么的？用一句话描述。"
```

如果 Phase 1 已经可以推断，改为确认式：
```
"根据我的探索，这个 API 是用来 {推断的用途}，对吗？"
```

### Q2: 目标用户

```
"谁会使用基于这个 API 的 skill？
 a) 技术人员（了解 API 参数）
 b) 业务人员（只会说自然语言）
 c) 两者都有"
```

### Q3: 典型用户表达

```
"请举 3-5 个例子，用户通常会怎么描述他们的需求？
 比如他们会说什么样的话来触发查询/操作？"
```

追问：
```
"这些例子中，最简单的请求长什么样？最复杂的呢？"
```

### Q4: 请求格式（如 Phase 1 未完全覆盖）

```
"API 请求体是什么格式？（JSON / GraphQL / SQL / 其他）
 有哪些必填参数？有哪些重要的选填参数？"
```

### Q5: 输出展示

```
"查询/操作的结果应该怎么展示给用户？
 a) 直接展示原始数据
 b) 先用自然语言解释，再展示数据
 c) 转换为表格/图表
 d) 其他格式"
```

如果选 b：
```
"解释部分需要包含哪些要素？（比如：查了什么、时间范围、计算方式…）"
```

---

## P1: 领域规则（推荐收集）

> 目标：收集提升 skill 质量的关键知识。
> 预计耗时：5-10 分钟

### Q6: 硬性规则（Iron Rules）

开场：
```
"使用这个 API 时，有没有那种'做了之后结果看起来正常但其实是错的'的陷阱？
 比如某个参数用错了不会报错，但返回的数据是错误的。"
```

对每条规则追问：
```
"能举一个正确做法和错误做法的具体例子吗？"
"这个错误在什么场景下最容易发生？"
```

循环：
```
"还有其他类似的陷阱吗？"
```

### Q7: 语义消歧

```
"用户说的话中，有没有同一个词在不同情境下含义不同的情况？
 比如用户说'XX'，有时候指 A，有时候指 B？"
```

追问：
```
"当用户没有说清楚时，应该默认理解为哪个含义？"
```

### Q8: 常见错误

```
"新手（或者 AI）使用这个 API 最常犯的错误是什么？
 列举你见过的前 3-5 个最常见的错误。"
```

对每个错误：
```
"这个错误的表现是什么？（用户会看到什么异常？）
 正确做法是什么？"
```

---

## P2: 高级知识（可选）

> 目标：锦上添花，使 skill 覆盖更多场景。
> 预计耗时：5-15 分钟

### Q9: 高级示例

```
"有没有比较复杂的使用场景？比如需要多步操作、嵌套查询或组合多个功能的情况？
 请举 2-3 个复杂案例。"
```

### Q10: 参数速查表

```
"有没有一些'用户常说的话 → API 参数'的固定映射？
 比如用户说'昨天'就一定对应某个特定的参数写法？
 列出最常用的 10-15 个这样的映射。"
```

### Q11: 默认策略

```
"当用户没有指定某个参数时，应该怎么自动推断？
 比如用户没说时间范围，应该默认查什么时段？
 有没有优先级规则？"
```

### Q12: 操作组合约束

```
"这个 API 的操作之间有没有组合限制？
 比如操作 A 和操作 B 不能同时使用，
 或者做了操作 A 之后不能直接做操作 B，必须分步？"
```

### Q13: 筛选条件路由

```
"如果 API 有多个放置筛选条件的位置（比如请求前过滤 vs 结果后过滤），
 什么情况下应该放在哪里？有没有决策规则？"
```

### Q14: 输出翻译

```
"API 返回的技术参数/字段名，需要翻译成用户能理解的业务语言吗？
 如果需要，主要有哪些翻译规则？"
```

## 访谈结束检查

每轮结束后，向用户展示收集进度：

```
📋 知识文档进度：

✅ 已收集：
  - 领域概述
  - 工具定义 ({N} 个)
  - 请求格式
  - 输出规范
  - 硬性规则 ({N} 条)

⬜ 待收集（推荐）：
  - 语义消歧规则
  - 完整示例（至少 3 个）
  - 错误模式

⬜ 待收集（可选）：
  - 参数速查表
  - 默认策略
  - 操作组合约束

下一步：
  a) 继续补充推荐项
  b) 先生成基础版知识文档
  c) 补充某个特定章节
```

## 追问技巧

### 当用户回答"没有"时

不要直接接受，用引导性问题确认：
```
"真的没有吗？让我换个方式问——
 如果一个完全不了解这个系统的人来用，
 他最可能犯什么错误？"
```

### 当用户给出模糊回答时

用具体化问题追问：
```
"能举一个具体的例子吗？
 比如用户说了'XX'，正确的处理方式是什么，错误的又是什么？"
```

### 当用户提供了大量信息时

及时结构化并确认：
```
"让我整理一下你说的——
 规则 1: {概括}
 规则 2: {概括}
 ...
 我的理解对吗？有没有遗漏？"
```
