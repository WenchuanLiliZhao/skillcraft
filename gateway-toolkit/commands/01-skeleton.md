### 类型说明

- **U (Universal)**: 任何 API 查询 skill 都需要，结构和内容均通用
- **D (Domain)**: 内容完全特定于此领域，但内部结构有可复用的子模式
- **E (Environment)**: 内容随运行环境（Cursor/Claude Code/Portal）而变
- **T (Template)**: 章节结构通用，但具体内容需按领域填充

## 三、D 类内容的子模式清单

以下 8 个子模式是从领域特定内容中提取的**通用结构**。任何领域的 skill 都可以用这些模式组织其领域知识。

### P1: 硬性规则模式 (Iron Rules Pattern)

**来源**: SKILL.md 铁律 0-9（第 35-121 行）

**通用结构**（每条规则按字段填写，顺序固定）:

| 字段 | 内容 |
|------|------|
| 标题行 | `规则 {编号} — {规则标题}` |
| 描述 | 1–3 句话说明规则 |
| 后果 | 违反时为何会产生错误结果 |
| 正确 | ✅ 正确做法 + 示例 |
| 错误 | ❌ 错误做法 + 示例 |
| 易错子场景 | ⚠️ 可选：特别容易踩坑的补充说明 |

**通用化描述**: 一组"违反即产生错误结果"的硬性约束。来源于实际使用中反复踩过的坑。数量通常 5-15 条。每条必须有正反示例。

**其他领域示例**:
- SQL 查询 skill: "GROUP BY 中必须包含 SELECT 中的所有非聚合列"
- REST API skill: "PUT 请求必须包含完整资源表示，不可只传部分字段"

---

### P2: 需求分解框架模式 (Decomposition Framework Pattern)

**来源**: SKILL.md §0 语义理解的四维度分解（第 125-126 行）

**通用结构**: 先写一句总述，再列维度表。

- **总述**: 将用户请求分解为 N 个正交维度（通常 3–5 个），每维对应一类 API 参数。

| 维度 | 核心问题 | 示例（占位） |
|------|----------|----------------|
| 维度1 | 用户要什么？ | … |
| 维度2 | 怎么处理？ | … |
| 维度3 | 针对谁？ | … |
| 维度4 | 什么范围？ | … |

**通用化描述**: 在处理用户请求前，先将模糊的自然语言拆解为几个正交维度。每个维度对应 API 的一类参数。维度数量通常 3-5 个。

**其他领域示例**:
- SQL 查询 skill: 查什么（SELECT）/ 从哪查（FROM+JOIN）/ 筛选什么（WHERE）/ 怎么聚合（GROUP BY）
- 报告生成 skill: 主题（关于什么）/ 受众（给谁看）/ 深度（概览 vs 详细）/ 格式（PDF/PPT/表格）

---

### P3: 语义消歧规则模式 (Disambiguation Rules Pattern)

**来源**: SKILL.md 规则 A-J（第 127-173 行）

**通用结构**:

- **标题行**: `规则 {字母} — {歧义点标题}`
- **映射表**（每种歧义一行）:

| 用户说 | 实际含义 | 实现方式 |
|--------|----------|----------|
| 「表达1」 | 含义A | 做法A |
| 「表达2」 | 含义B | 做法B |
| 「表达3」（无修饰） | 歧义，默认 {X} | 做法X |

**通用化描述**: 用户自然语言中存在歧义的表达，需要明确的映射规则。每条规则处理一类歧义，给出判断标准和默认选择。数量通常 5-15 条。

**其他领域示例**:
- SQL skill: "最近" = 最近 7 天还是最近 30 天？（默认 7 天）
- 搜索 API skill: "包含" = 精确包含还是模糊匹配？（默认模糊）

---

### P4: 参数速查表模式 (Quick Reference Table Pattern)

**来源**: SKILL.md timeConstraint 速查表（第 414-417 行）、默认策略表（第 419-421 行）

**通用结构**: 自然语言 → 可复制 API 参数值的速查表（建议覆盖 10–20 种常见表达）。

| 用户说 | 对应 API 参数值 |
|--------|-------------------|
| {自然语言表达1} | {精确的 API 参数写法} |
| {自然语言表达2} | {精确的 API 参数写法} |
| … | … |

**通用化描述**: 将最常见的用户表达直接映射为可复制的 API 参数值。作用是让 AI 不需要临场推导，减少出错。

**其他领域示例**:
- SQL skill: "昨天" → `WHERE date = CURRENT_DATE - 1`
- Elasticsearch skill: "精确匹配" → `{"term": {"field": "value"}}`

---

### P5: 默认策略决策树模式 (Default Strategy Decision Tree Pattern)

**来源**: SKILL.md 用户未指定时间时的默认策略（第 419-422 行）

**通用结构**:

1. **前提**: 当用户未指定 `{参数}` 时，按下面优先级匹配（命中即停）。
2. **决策表**:

| 优先级 | 条件 | 默认值 | 原因 |
|--------|------|--------|------|
| 1 | 查询含 X 特征 | 默认值A | … |
| 2 | 查询含 Y 特征 | 默认值B | … |
| … | … | … | … |
| N | 其他 | 默认值Z | 兜底 |

**通用化描述**: 当用户省略某个参数时，根据查询上下文自动推断合理的默认值。使用优先级决策树结构，避免歧义。

**其他领域示例**:
- SQL skill: 未指定排序 → 有聚合则按聚合值 DESC，无聚合则按主键 ASC
- 搜索 API skill: 未指定返回数量 → 有分页则 20，无分页则 100

---

### P6: 筛选条件路由模式 (Filter Routing Pattern)

**来源**: SKILL.md §3.4 filters vs resultFilters 决策树（第 390-409 行）+ 构建流程 Step 5（第 482-496 行）

**通用结构**（决策树，用文字层级表达）:

- **根问题**: 需要筛选吗？若需要，继续分支。
- **分支**:
  - 若 **条件 A** 成立 → 放在 **{位置1}**（说明：影响计算 / 不影响计算）
  - 若 **条件 B** 成立 → 放在 **{位置2}**
  - **其他** → 放在 **{位置3}**

**通用化描述**: API 有多个可以放置筛选条件的位置，每个位置的语义不同（前置过滤 vs 后置过滤）。需要决策树指导 AI 放对位置。

**其他领域示例**:
- SQL skill: WHERE vs HAVING（聚合前 vs 聚合后）
- GraphQL skill: 参数级过滤 vs 响应级过滤

---

### P7: 操作组合约束模式 (Composition Constraint Pattern)

**来源**: SKILL.md 铁律 5（不可链式叠加）+ §3.2.4 indirections 组合规则（第 77-91、366-373 行）

**通用结构**: 针对「操作 X」与「操作 Y」列出允许的组合方式。

| 组合方式 | 是否允许 | 正确做法 |
|----------|----------|----------|
| X 后直接接 Y | ❌ | 先定义中间变量，再对中间变量做 Y |
| X 和 Y 并列 | ✅ | 直接写 |
| Y 嵌套在 X 中 | 视 API 而定 | … |

**通用化描述**: API 的操作之间存在组合约束——某些操作可以叠加，某些不能。需要明确列出禁止的组合和替代做法。

**其他领域示例**:
- SQL skill: 聚合函数不能嵌套（`SUM(AVG(x))` 非法），需子查询分步
- 正则表达式 skill: 某些量词不能组合（`(a+)+` 导致灾难性回溯）

---

### P8: 输出翻译规则模式 (Output Translation Rules Pattern)

**来源**: SKILL.md §9.1 查询解读的翻译规则速查（第 555-558 行）

**通用结构**: 技术对象到业务语言的翻译表。

| 技术对象 | 翻译规则 | 示例 |
|----------|----------|------|
| {API 参数类型1} | {如何转化为业务语言} | {技术值} → 「{业务表达}」 |
| {API 参数类型2} | {如何转化为业务语言} | … |

**通用化描述**: 将 API 的技术参数翻译为非技术用户可理解的业务语言。每类参数都有固定的翻译模式。

**其他领域示例**:
- SQL skill: `INNER JOIN` → "只取两张表都有的记录"
- 搜索 API skill: `{"bool":{"must":...}}` → "同时满足以下所有条件"

## 四、内容分布统计

基于 SKILL.md 的 949 行（含空行）:

| 类型 | 行数 | 占比 | 说明 |
|------|------|------|------|
| **U (Universal)** | ~50 行 | 5% | 构建流程框架（步骤 0→1→2→3 的通用结构）+ 三段式输出规范 |
| **D (Domain)** | ~178 行 | 19% | 铁律(87) + 语义消歧(51) + 领域特定默认值/约束(40) |
| **E (Environment)** | ~60 行 | 6% | 工具调用方式(20) + 工具返回格式解析(40) |
| **T (Template)** | ~661 行 | 70% | 参数规范(177) + 示例(248) + 输出格式模板(143) + 错误模式(28) + 其余(65) |

### 关键洞察

1. **70% 是模板内容 (T)** — 结构通用但内容领域特定。这是 Compile 命令的主要工作：往模板里填充领域知识。

2. **19% 是不可替代的领域知识 (D)** — 铁律和消歧规则，只能由人提供。这是 Prepare 命令的核心收集目标。

3. **仅 6% 是环境特定的 (E)** — 工具调用方式。通过环境适配器可以完全自动化处理。

4. **仅 5% 是纯通用的 (U)** — 但这 5% 定义了整个 skill 的"骨架"，是最重要的结构性内容。

## 五、骨架的适用范围

此骨架适用于以下类型的 skill:

| 适用 | 不完全适用 |
|------|----------|
| API 数据查询（指标、SQL、搜索） | 纯代码生成（无 API 交互） |
| 结构化请求体构建（JSON、GraphQL） | 自由文本创作（写作、翻译） |
| 有明确输入→输出映射的工具操作 | 多轮对话式交互（客服、咨询） |
| 有"正确/错误"之分的操作 | 主观判断类任务（设计评审） |

对于不完全适用的场景，骨架中的部分章节仍可复用（如硬性规则模式、输出格式规范），但整体流程需要调整。
